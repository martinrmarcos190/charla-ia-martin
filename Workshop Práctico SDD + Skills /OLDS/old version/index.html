Bloque 1 · Fundamentos

Spec-Driven Development

  • El vibe coding no escala: el código parece bien, pero no anda o resuelve otra cosa. Se pierde la intención.
  • SDD da vuelta la jerarquía: la spec es la única fuente de verdad; el código es el output que se regenera.
Para un MLOps es Infrastructure-as-Code, pero para el código: no parcheás el server a mano, cambiás el Terraform y reaplicás. En SDD no parcheás el código: cambiás la spec y regenerás.
La analogía IaC es la que tiene que pegar. "Debuggear = arreglar la spec. Refactorizar = reescribir la spec para más claridad." Repetir: lo que gobierna es la spec.

Bloque 1 · Fundamentos

  • Toolkit open-source de GitHub que materializa SDD para agentes de código
  • CLI specify · bootstrapea tu repo con plantillas y slash-commands
  • Última release estable 0.8.18 (29-may-2026) · 0.9.0 en desarrollo

El flujo tiene fases con checkpoints:

Specify → Plan → Tasks → el qué/porqué, el cómo técnico, el desglose
Implement → la construcción · vos manejás el volante, el agente escribe
"Spec Kit le da estructura al SDD: en vez de un prompt suelto, un flujo con gates." Hoy lo corremos entero. Pinneá versión si querés que todos vean lo mismo.

Recorré cada comando: qué hace y qué produce. clarify y analyze son los gates de calidad. La diferencia con vibe coding es justamente que hay checkpoints revisables entre fase y fase.

Bloque 1 · Spec Kit en detalle

Dónde viven los artefactos

.specify/ └── memory/ └── constitution.md # principios del proyecto .claude/ └── commands/ # los slash-commands /speckit.* specs/ └── 001-inventario/ ├── spec.md # qué construir ├── plan.md # cómo (tech stack) └── tasks.md # lista ejecutable

Todo se commitea. La spec, el plan y las tareas viven en el repo, versionados, revisables en un PR — igual que cualquier otra parte del sistema.

El punto fuerte para MLOps: estos artefactos son code-reviewable y versionables. Cuando alguien pregunta "¿por qué el código hace X?", la respuesta está en spec.md, no en un chat perdido.

Bloque 4 · El objetivo

Un MCP propio + una Skill, vía SDD

Pedido del usuario  ("listá los modelos y agregá uno")
  └─→ Skill "items-ops"        (el procedimiento: cómo operar el inventario)
        └─→ MCP "items-api"     (las tools: list_models, get_model, add_model)
              └─→ API Flask     (GET/POST → SQLite)   ✓ ejecutado
MCP aporta la conectividad — las tools hacia tu API
Skill aporta el procedimiento — cómo usarlas bien
Mostrá el objetivo end-to-end primero. La Skill no "habla" con la API: le dice al modelo CÓMO usar las tools que el MCP expone. Ese es el matrimonio MCP + Skills.

Bloque 4 · El MCP · SDK oficial

MCP server mínimo (FastMCP, stdio)

from mcp.server.fastmcp import FastMCP
import httpx, logging, sys

# stdio: NUNCA print() a stdout (corrompe el JSON-RPC)
logging.basicConfig(level=logging.INFO, stream=sys.stderr)

mcp = FastMCP("items-api")
API_BASE = "http://127.0.0.1:5000"

@mcp.tool()
async def list_models() -> str:
    """Lista todos los modelos del inventario local."""
    async with httpx.AsyncClient() as c:
        r = await c.get(f"{API_BASE}/models", timeout=30.0)
        r.raise_for_status()
        return str(r.json())

# ... get_model(model_id) y add_model(name, framework, accuracy)

if __name__ == "__main__":
    mcp.run(transport="stdio")

SDK oficial mcp 1.27.x · Python ≥3.10 · setup con uv add "mcp[cli]" httpx

El gotcha crítico: en stdio, print() a stdout rompe el protocolo JSON-RPC. Por eso logging va a stderr. El código completo está en entregables/mcp-items/server.py. Cada tool se describe sola con el docstring.

Bloque 4 · El artefacto

El SKILL.md resultante

---
name: items-ops
description: Opera el inventario de modelos (listar, consultar
  y dar de alta) a través del MCP items-api. Usar cuando el
  usuario pida ver, buscar o agregar modelos al inventario.
---

# items-ops
## Reglas
- Usá siempre los nombres calificados de las tools:
  items-api:list_models · items-api:get_model · items-api:add_model
- Antes de un alta, validá que accuracy ∈ [0, 1].
- Después de un alta, confirmá el id con items-api:get_model.
  • Frontmatter: name (≤64, minúsculas/guiones) + description (≤1024, 3ª persona, qué + cuándo)
  • Progressive disclosure: solo se carga name+description hasta que la tarea matchea · cuerpo <500 líneas
Subrayá las dos reglas de oro del frontmatter. La buena práctica de nombres calificados servidor:tool evita errores "tool not found" cuando hay varios MCP. El description es lo único que Claude lee al arrancar — tiene que decir cuándo usarse.

Bloque 5 · Reflexión

Skills + MCPs + imaginación

MCP te da la capacidad (las tools hacia tus sistemas). Skill te da el criterio (cómo se hace acá). Combinándolos, construís casi cualquier integración.
"MCP can provide tools and resources, while Skills package the playbook for using them well." — Docs de Claude (Agent Skills)
La frase que tiene que quedar: MCP = capacidad, Skill = criterio. Con esas dos piezas y un poco de imaginación, el agente se vuelve una capa más de tu stack de DevEx.

Bloque 5 · Ejemplos

Skills útiles para MLOps

Mostrá amplitud: de lanzar entrenamientos a generar model cards. El patrón run-training refleja el run-skill-generator oficial. Los scripts bundleados dan confiabilidad sin gastar contexto.

Bloque 5 · La idea central

Diseñá tu propio SDD basado en Skills

El flujo specify→plan→tasks→implement no es fijo. Como una Skill es solo una carpeta con un procedimiento, encodeá cada fase a medida de tu stack:

mlops-specify → tus specs siempre con sección de datos, métricas objetivo y latencia
mlops-plan → impone tu stack: Kubeflow, MLflow, tu registry
mlops-tasks → genera tareas con tus convenciones de CI
mlops-implement → conoce tus repos y patrones

Tu equipo deja de usar un SDD genérico y pasa a tener su propio SDD, versionado en el repo y compartible vía plugins.

El cierre conceptual fuerte. Conectá con el ejercicio del Bloque 1 (las 3 reglas de su constitución). MCP = conectividad, Skills = procedimiento; juntos = el SDD de TU equipo, no uno genérico.
