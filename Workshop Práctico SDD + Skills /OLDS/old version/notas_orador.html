Slide 4 · Bloque 1

Spec-Driven Development

⏱️ 00:10–00:14 · Bloque 1 — Fundamentos (25 min)

💬 Qué decir

  • Vibe coding: el código parece bien pero no anda o resuelve otra cosa. Se pierde la intención.
  • SDD da vuelta la jerarquía: la spec es la única fuente de verdad; el código se regenera.
  • La analogía que tiene que pegar: IaC, pero para el código.

🎯 No se pueden ir sin entender

"En SDD no parcheás el código: cambiás la spec y regenerás. Debuggear es arreglar la spec."

📜 Guion completo · 1 · Qué es SDD

"Cuando le tirás un prompt suelto a un agente —'hacéme una API de inventario'— pasa lo de siempre: el código parece correcto, pero no compila, o resuelve otra cosa, o elige un stack que no es el tuyo. Eso es vibe coding, y está bien para un prototipo del viernes, pero no escala. El problema no es el modelo: es que lo tratamos como un buscador cuando es un programador de a pares muy literal que necesita instrucciones sin ambigüedad.

SDD da vuelta la jerarquía: en vez de que la spec sirva al código, el código sirve a la spec. La especificación es la única fuente de verdad, y el código pasa a ser el output que se regenera. Para ustedes, MLOps, esto es familiar: es Infrastructure-as-Code, pero para el código. Igual que no editás un server a mano sino que cambiás el Terraform y reaplicás, en SDD no parcheás el código: cambiás la spec y regenerás. Debuggear es arreglar la spec; refactorizar es reescribir para que la spec sea más clara."

📜 Guion completo · 1 · Qué es Spec Kit

"Spec Kit es el toolkit open-source de GitHub que materializa SDD para agentes de código —Claude Code, Copilot, Gemini y otros—. Trae una CLI llamada specify que bootstrapea tu repo con plantillas y slash-commands. El flujo tiene fases con checkpoints: primero una constitución con los principios no-negociables del proyecto, después specify —el qué y el porqué—, plan —el cómo técnico—, tasks —el desglose accionable— e implement —la construcción—. La idea es que vos manejás el volante y el agente escribe. Hoy lo vamos a correr entero sobre un problema real."

Slide 6 · Bloque 1

El flujo de comandos /speckit.*

⏱️ 00:17–00:22 · Spec Kit en detalle

💬 Qué decir (comando por comando)

  • constitution → principios · specify → el qué · clarify (opc) → ambigüedades
  • plan → el cómo técnico · tasks → desglose · analyze (opc) → gate de consistencia · implement → construye
  • clarify y analyze son los gates de calidad — eso es lo que NO tiene el vibe coding.

🎯 Ojo con los nombres

Hoy llevan prefijo: /speckit.specify, etc. El flag de init es --integration claude; el viejo --ai quedó deprecado.

📜 Guion completo · 1 · El flujo en detalle

"Veamos el flujo comando por comando, porque en un rato lo vamos a tipear. /speckit.constitution sienta los principios no-negociables. /speckit.specify describe qué construir y por qué, sin entrar en el cómo. /speckit.clarify —opcional pero recomendado— hace preguntas para cerrar ambigüedades, y acá es donde gobernás la intención. /speckit.plan define el cómo técnico, el stack. /speckit.tasks lo baja a un desglose accionable. /speckit.analyze —otro gate— chequea que spec, plan y tasks no se contradigan. Y /speckit.implement construye. La diferencia con el vibe coding son justamente esos checkpoints revisables entre fase y fase. Y un detalle importante: los nombres cambiaron, hoy llevan el prefijo speckit., y el flag de init es --integration claude, no el viejo --ai."

Lo importante para ustedes: todo esto se commitea. La spec, el plan y las tareas viven en el repo, versionados y revisables en un PR, igual que cualquier otra parte del sistema. Cuando dentro de seis meses alguien pregunte por qué el código hace algo, la respuesta está en la spec, no en un chat que ya nadie tiene."

Slide 18 · Bloque 4

MCP server mínimo (FastMCP, stdio)

⏱️ 01:48–01:52

💬 Qué decir

  • SDK oficial mcp (1.27.x, Python ≥3.10). from mcp.server.fastmcp import FastMCP.
  • Cada tool se describe sola con el docstring — eso es lo que el modelo lee.
  • Código completo en entregables/mcp-items/server.py.

🎯 El gotcha crítico

En stdio, NUNCA print() a stdout → corrompe el JSON-RPC. Por eso el logging va a sys.stderr.

📜 Guion completo · 4 · El código del MCP

"Este es el MCP server mínimo, con el SDK oficial. Importás FastMCP, creás el server con un nombre, y cada función que decorás con arroba mcp punto tool se vuelve una tool invocable. Fíjense que cada tool tiene un docstring: ese docstring es exactamente lo que el modelo lee para decidir cuándo llamarla. Tenemos tres: listar modelos, traer uno por id, y agregar uno, todas pegándole a la API Flask con httpx.

Y acá va el gotcha más importante de todos, subráyenlo: en transporte stdio, nunca escriban a stdout. Un print corrompe el JSON-RPC y el server deja de funcionar. Por eso configuramos el logging hacia stderr. El código completo está en los entregables, no hace falta que lo copien letra por letra ahora."

Slide 21 · Bloque 4

El SKILL.md resultante

⏱️ 02:12–02:15

💬 Qué decir (las reglas de oro)

  • name: ≤64 chars, minúsculas/números/guiones. description: ≤1024, 3ª persona, qué + cuándo.
  • Nombres calificados servidor:tool → evita "tool not found" con varios MCP.
  • Progressive disclosure: al arrancar solo se carga name+description; cuerpo <500 líneas.

🎯 Lo único que Claude lee al arrancar

Es el description. Por eso tiene que decir qué hace y cuándo usarse — si está mal escrito, la Skill no se activa nunca.

📜 Guion completo · 4 · Anatomía del SKILL.md

"Miremos el archivo que generó. Arriba, entre tres guiones, está el frontmatter YAML. Dos campos obligatorios: name —máximo 64 caracteres, solo minúsculas, números y guiones— y description —máximo 1024 caracteres, en tercera persona, diciendo qué hace y cuándo usarse—. Ese description es lo único que Claude lee al arrancar la sesión. Si está mal escrito, la Skill no se activa nunca. Por eso dice explícitamente 'usar cuando el usuario pida ver, buscar o agregar modelos'.

En el cuerpo, las reglas: usar siempre los nombres calificados items-api dos puntos list_models, etcétera —eso evita errores de 'tool not found' cuando hay varios MCP conectados—, validar accuracy entre cero y uno antes de un alta, y confirmar el id después. Y todo esto con progressive disclosure: al inicio solo se cargan el nombre y la descripción; el cuerpo se lee recién cuando la tarea matchea. Por eso el cuerpo se mantiene por debajo de 500 líneas."

Slide 24 · Bloque 5

Skills útiles para MLOps

⏱️ 02:23–02:25

💬 Qué decir (rápido, 2-3 ejemplos)

  • run-training: captura la receta reproducible de lanzar un entrenamiento (env, vars, script).
  • data-drift-check: corre un drift.py bundleado e interpreta el reporte. Script determinístico = confiable.
  • model-card: genera la model card con el template del equipo, tomando métricas del MCP.
Patrón: cualquier procedimiento repetible que quieras que el agente haga siempre igual. El run-training refleja el run-skill-generator oficial.
📜 Guion completo · 5 · Ejemplos para su mundo

"Para que lo aterricen en su mundo, tres ejemplos de Skills MLOps. Una run-training que captura el procedimiento reproducible de levantar un entrenamiento —los comandos de entorno, las variables, el script de lanzamiento que de verdad funcionó— y queda commiteada para que todo el equipo siga la misma receta. Una data-drift-check que corre un script de drift bundleado entre el dataset de referencia y producción, e interpreta el reporte; al ser un script determinístico, es confiable y no gasta contexto. Y una model-card que genera la model card siguiendo el template del equipo, tomando las métricas del inventario vía el MCP. El patrón es siempre el mismo: cualquier procedimiento repetible que quieras que el agente haga siempre igual."
