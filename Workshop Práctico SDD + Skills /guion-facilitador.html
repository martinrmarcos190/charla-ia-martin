Taller práctico: SDD + Skills para DevOps

El caso: una API local de issues de infraestructura, un MCP que la expone, una skill que opera los issues y analiza logs, otra que genera el reporte HTML, y todo empaquetado en un plugin/Power.

Reparto: B0 apertura 10' · B1 Fundamentos 25' · B2 Setup 25' · B3 API de issues 45' · B4 MCP + Skill 35' · B5 Análisis de logs 20' · B6 Reporte + plugin 25' · B7 Reflexión 10'. Total ~195 min (núcleo B0-B4: 150'; B5/B6 son elásticos — con slot de 2 h el taller cierra digno al final del B4). Donde hay divergencia se marca el camino.

Tené siempre a mano la tabla maestra de equivalencias en pantalla (slide 8 del Bloque 1) y el 4.G de troubleshooting (al final de este guion).

  • El cron asesino (periodicidad): un analytics-rollup horario en infra satura db-01 → acquires lentos en payments → 504s en el gateway, cada hora en punto. Actualiza el issue de payments.
  • Leak post-deploy (tendencia): deploy de checkout v2.8.0 a las 07:14 → el pool crece 7→50 en INFOs a lo largo de 3 horas → 502s recién a las 10:41. Actualiza el issue de 502s.
  • Clock skew → JWT (ventana): corrección NTP de -2.4s en worker-2 → "token used before issued" → ráfaga de 401s de 11 minutos. Crea issue nuevo.
  • Disco + backups (la fácil, para entrar en calor): disco 84→97% → backups fallan. Actualiza el issue de backups.
El momento clave para narrar: cuando la skill cite líneas de DOS o TRES archivos distintos como cadena causal (cron INFO + acquire WARN + 504) — eso es lo que ningún grep hace. Y que descarte los 37 errores de image-resize con justificación. Cierran verificando con curl /issues.

4.G

Troubleshooting

Comunes
  • Python viejo/no encontrado: python3 --version; si <3.11, uv python install 3.12. Spec Kit y el SDK MCP requieren ≥3.10.
  • uv/uvx "command not found": reabrí la terminal; si persiste, agregá ~/.local/bin al PATH.
  • MCP 0 tools / no conecta: corré el server a mano para ver errores en stderr; nunca print() a stdout (corrompe el JSON-RPC); usá ruta absoluta en el working directory; el server stdio descubre tools al inicio de sesión.
  • Flask/SQLite: "no such table" → no inicializaste la DB; POST falla → falta -H "Content-Type: application/json"; puerto 5000 ocupado → cambialo.
  • macOS + puerto 5000 (¡frecuente!): AirPlay Receiver escucha en el 5000 y responde 403 con body vacío — parece que el server "no anda" pero es AirPlay. Desactivar en Ajustes del Sistema → General → AirDrop y Handoff → Receptor de AirPlay, o usar otro puerto. Diagnóstico: lsof -nP -iTCP:5000 -sTCP:LISTEN (si aparece ControlCe, es AirPlay).
  • Flask viejo ('Flask' object has no attribute 'get'): uv run --with flask agarró un Python viejo del sistema. Fix: uv run --python 3.12 --with flask app.py.
A Camino A · Claude Code + Spec Kit
  • Claude Code "command not found": abrí terminal nueva o source ~/.zshrc; el binario nativo está en ~/.local/bin/claude.
  • Node viejo (instalación npm legada): requiere Node 18+; mejor el instalador nativo; nunca sudo npm.
  • No aparecen los /speckit.*: carpeta equivocada o no reiniciaste; ls .claude/commands/, reiniciá Claude Code; usá --integration claude (no el viejo --ai).
  • MCP no aparece tras claude mcp add: claude mcp list y abrí sesión nueva (descubre tools al inicio).
B Camino B · Kiro
  • No veo el panel Specs: confirmá que abriste una carpeta (no un archivo suelto); reabrí con File → Open Folder.
  • La Skill no se reconoce: debe estar en .kiro/skills/<nombre>/SKILL.md (subcarpeta con el nombre), no suelta bajo skills/; revisá el frontmatter (name, description).
  • El MCP no conecta desde mcp.json: validá el JSON (comas, llaves); ruta absoluta en --directory; verificá que el Flask del Bloque 3 esté corriendo; mirá los logs del panel MCP; referenciá variables como ${VAR} y no hardcodees secretos.
  • El Feature Spec no avanza: en Requirements-First hay gates entre fases; para la demo, usá Quick Plan para saltarlos.
B5/B6 Logs, reporte y plugin
  • La skill no encuentra los logs: pasale la ruta absoluta de la carpeta logs/ (o copiala dentro del proyecto del taller); verificá permisos de lectura.
  • El análisis se queda solo con los ERRORs (cae en los red herrings): la spec pide buscar periodicidad/tendencias/ventanas aunque no haya ERROR — si el SKILL.md regenerado perdió eso, repetí implement; como pista en vivo: "¿qué pasa cada hora en punto?".
  • El análisis duplica issues en vez de actualizar: la spec exige comparar con list_issues antes de crear — si duplica, revisá que el SKILL.md regenerado conserve esa regla y repetí implement.
  • El reporte sale vacío o viejo: la skill debe llamar a issues-api:list_issues en el momento — si "recuerda" datos de antes, pedile explícito "consultá el MCP de nuevo"; verificá que el Flask siga corriendo.
  • El plugin no aparece (A): estructura exacta .claude-plugin/plugin.json + skills/<nombre>/SKILL.md (¡skills/ va en la raíz del plugin, NO adentro de .claude-plugin/!); tras instalar, /reload-plugins; /plugin para ver el estado; atajo para probar: claude --plugin-dir ./devops-issues.
  • El Power no toma las skills (B): depende de la versión — dejá las skills en .kiro/skills/ y el Power con MCP + steering; el mensaje pedagógico no cambia.

  1. Antes: mandá el material previo 3–4 días antes; pedí el output del checklist en un canal.
  2. Durante: mostrá la tabla maestra de equivalencias todo el tiempo.
