Reflexión y extensión

De armar un taller a armar tu propio SDD.

La ecuación

Skills + MCPs + imaginación = construís cualquier cosa
MCP = conectividad (las tools, el acceso a tus sistemas).
Skills = procedimiento (cómo se hace ACÁ: tus reglas y pasos).
Son complementarios: una sin la otra queda coja.
Dos piezas, dos roles

MCP vs Skills

MCP = conectividad

  • Tools hacia APIs, DBs, servicios.
  • Define qué puede tocar el agente.
  • El "cableado" hacia tus sistemas.

Skills = procedimiento

  • Receta reproducible, progressive disclosure.
  • Define cómo lo hace tu equipo y cuándo.
  • El "manual de operación".
Hoy lo hicieron entero

El patrón que acaban de construir

API de issues → MCP issues-api → skill issues-ops (opera y lee logs) → skill issues-report → plugin devops-issues.

El mismo patrón aplica a cualquier flujo de tu equipo:

incident-triage

Recibe la alerta, junta contexto (logs, métricas, deploys recientes vía MCPs), clasifica severidad según TU runbook y abre el incidente con el template del equipo.

runbook-executor

Encodea el runbook de operaciones (rotar credenciales, escalar un servicio, drenar un nodo) como procedimiento verificable, paso a paso, con confirmaciones.

postmortem-writer

Genera el postmortem desde la línea de tiempo del incidente (issues + logs vía MCP), siguiendo el formato del equipo: impacto, causa raíz, acciones.

En Kiro funcionan igual (mismo SKILL.md en .kiro/skills/). Si una necesita conectividad además del procedimiento, empaquetala como Power.

El giro final

Diseñá tu propio SDD basado en Skills

El flujo specify → plan → tasks → implement no es fijo.

Una Skill es solo una carpeta con un SKILL.md que enseña un procedimiento. Encodeá cada fase a medida de tu stack:

  • specify → sabe que tus specs llevan SLOs, plan de rollback y criterios de monitoreo.
  • plan → impone tu stack (Terraform, ArgoCD, tu observabilidad).
  • tasks → con tus convenciones de CI/CD y ventanas de deploy.
  • implement → conoce tus repos, módulos y patrones.
Cómo lo armás en cada herramienta

Las piezas para tu SDD de equipo

A Claude Code + Spec Kit

  • Skills → .claude/skills/
  • Constitución → .specify/memory/
  • Compartible vía plugins — como el devops-issues que armaron en el B6.

B Kiro

  • Skills → .kiro/skills/
  • Steering files → .kiro/steering/
  • Powers: empaquetan MCP + steering + hooks, instalables desde la UI.
Solo en Kiro · lo que acaban de hacer

Powers

Un Power es un superset: empaqueta MCP servers + steering + hooks en una unidad distribuible al equipo (frontmatter POWER.md).

En el B6 empaquetaron devops-issues: ese es el camino para distribuir el toolkit al resto del equipo.

La conclusión

Dejás de usar un SDD genérico y pasás a tener el tuyo: a medida del stack y portable entre herramientas.
