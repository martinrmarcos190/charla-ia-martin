Qué es SDD y por qué

El "vibe coding" no escala

El problema

  • Prompts sueltos, resultados irreproducibles.
  • Nadie sabe qué se pidió ni por qué.
  • Imposible de revisar o versionar como intención.

La idea de SDD

  • La spec es la fuente de verdad.
  • El código es un artefacto de build.
  • Reproducible, versionable, revisable.
SDD es a tu código lo que Infrastructure-as-Code es a tu infraestructura: declarás el QUÉ, la herramienta genera el CÓMO.
El flujo en abstracto

Las fases del SDD

constitución / reglas→ specify→ clarify→ plan→ tasks→ analyze→ implement
  • specify = el QUÉ · clarify = cerrar ambigüedades · plan = el CÓMO técnico.
  • tasks = descomponer · analyze = gate de consistencia · implement = generar el código.
  • Entre fases hay gates de aprobación humana: vos revisás antes de avanzar.
La idea de fondo · anti lock-in

SDD, MCP y Skills son estándares abiertos

El .md del problema, el server MCP en Python y el SKILL.md son idénticos en las dos herramientas.

Solo cambia DÓNDE se registran, no QUÉ son.

El 90% del taller es igual vengas con el Camino A o el Camino B.
