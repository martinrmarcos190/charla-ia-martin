2.3 · MCPs y Tools

Primero: ¿cómo usa una tool un LLM?

⏱️ 00:43–00:46 · Bloque 2.3 (≈20 min) · post-break

💬 Qué decir

  • Un LLM solo genera texto. Para que actúe le das tools: una función + su descripción.
  • El modelo no ejecuta nada: decide qué tool llamar y con qué argumentos. Tu código la corre y le devuelve el resultado.
  • Ejemplo de la slide (LangChain): get_weather bindeada al modelo → el modelo pide get_weather("Córdoba") → tu runtime la ejecuta.

🎯 No se pueden ir sin entender

El modelo decide, tu código ejecuta. Definir esto a mano para cada modelo y framework no escala → eso es lo que MCP estandariza. Puente directo al M×N.

📜 Guion completo · 2.3 · Cómo usa una tool un LLM

"Antes de MCP, una pregunta más básica: ¿cómo hace un LLM para usar una herramienta? Un LLM, solo, genera texto. Para que actúe le das tools: una función con su descripción. Y ojo con esto — el modelo no ejecuta nada. El modelo decide qué tool llamar y con qué argumentos; tu código la corre y le devuelve el resultado.

Acá lo ven con LangChain: defino get_weather con su docstring, se la bindeo al modelo, y cuando pregunto '¿qué tiempo hace en Córdoba?' el modelo me pide get_weather("Córdoba") y mi runtime la ejecuta. Ahora bien: definir esto a mano para cada modelo y cada framework no escala. Eso es exactamente lo que MCP estandariza — y nos lleva al problema M×N."

2.3 · MCPs y Tools

Las 3 primitivas de MCP

⏱️ 00:51–00:54

💬 Qué decir (mapear a algo que ya conocen)

  • Tools → funciones invocables por el modelo (query DB, mandar email, crear issue).
  • Resources → data leíble por el modelo (archivos, configs, registros · por URI (identificador de recurso)).
  • Prompts → templates reutilizables / slash commands.
Handshake: initialize → capabilities → tools/list → tools/call.
📜 Guion completo · 2.3 · Las 3 primitivas

"Un servidor MCP expone tres primitivas:

  • Tools — funciones que el modelo puede invocar (query a DB, mandar email, crear issue). El modelo decide cuándo llamarlas. Las describe con name, description y JSON Schema de inputs.
  • Resources — data que el modelo puede leer (archivos, registros, configs). Identificadas por URI.
  • Prompts — templates reutilizables que el host ofrece como slash commands.

El handshake: el cliente arranca, llama initialize, el servidor responde con sus capabilities, el cliente lista las tools con tools/list, el modelo decide qué invocar con tools/call. Para los que vienen de gRPC, es lo mismo: contract-first, capability discovery, request/response tipado."

Esto se llama 'progressive disclosure': cero overhead de contexto hasta que el skill se necesita de verdad."
