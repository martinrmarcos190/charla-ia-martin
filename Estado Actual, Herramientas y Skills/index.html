2.2 · Vibe coding vs SDD

Vibe coding

"There's a new kind of coding I call vibe coding, where you fully give in to the vibes, embrace exponentials, and forget that the code even exists." — Andrej Karpathy, 2 de febrero de 2025
  • Collins Dictionary Word of the Year 2025 (6-nov-2025)
  • = deploy a mano por SSH a producción
  • OK para el prototipo del viernes. NO para tu pipeline de CI.
La analogía del SSH es la que tiene que pegar.

2.2 · Vibe coding vs SDD

Por qué vibe coding no escala

  • El código co-escrito con IA trae 1.4× más issues críticos y 1.7× más issues mayores que el código humano
  • 45% del código generado por IA introduce vulnerabilidades del OWASP Top 10
  • La mayoría de los modelos no mejoran la seguridad del código que generan
  • Falta lo de siempre: reproducibilidad, revisión, versión, rollback
Datos crudos, fuerte y rápido. No te quedes en los números, llevá a "falta proceso".

2.2 · Vibe coding vs SDD

Spec-Driven Development (SDD)

  • Escribís una spec de qué querés; el agente la convierte en plan, tareas y código
  • Lo que gobierna es la spec, no el código generado
Pasamos de el código es la fuente de verdad a la intención es la fuente de verdad: la IA hace que las especificaciones sean ejecutables.
La frase clave: la spec se volvió ejecutable. Repetir "lo que gobierna es la spec".

2.2 · Vibe coding vs SDD

Spec Kit en una slide

  • Toolkit open-source de GitHub para SDD (sep-2025)
  • github/spec-kit · ⚠️ ~92k stars · 8k+ forks
  • Soporta 30+ agentes (Claude Code, Copilot, Gemini, Cursor, Codex, Windsurf, Kiro…)

Flujo canónico (slash commands):

/speckit.constitution   → principios no-negociables
/speckit.specify        → qué construir (requirements)
/speckit.clarify        → resolver ambigüedades
/speckit.plan           → plan técnico (tech stack)
/speckit.tasks          → lista de tareas ejecutable
/speckit.analyze        → consistencia entre artefactos
/speckit.implement      → ejecutar las tareas
La próxima clase se usa en vivo. Hoy solo la intuición.

🎬 Demo 1 · Vibe coding → "ahora escribime la spec"

El agente que vibe-codeó te da la spec

claude — ~/proyecto
$
> Hacé un endpoint Flask que reciba un POST con JSON, lo valide, lo guarde en SQLite y devuelva un ID.
✓ Creado app.py (Flask + sqlite3) ✓ Creado schema.sql ✓ Endpoint POST /items listo. Corriendo en :5000
# Funciona... pero vos no leíste nada. No es reproducible.
> Ahora generame la spec de lo que acabás de hacer: requirements, edge cases, dependencias y supuestos. Markdown.
# Spec — POST /items ## Requirements - Recibe JSON {name: str, qty: int} - Valida tipos; 400 si falta campo - Persiste en SQLite (tabla items) - Devuelve {id} con 201 ## Edge cases - Body vacío / no-JSON → 400 - qty negativo → 422 ## Dependencias: Flask 3.x, sqlite3 (stdlib)
El mismo agente que vibe-codeó te da el artefacto que necesitabas al principio. Si el agente puede escribir la spec después de programar, también puede escribirla antes y dejar que la spec dirija todo el desarrollo → reproducible, revisable, versionable.
Si la spec puede generarse al final, puede dirigir desde el inicio. Esa spec.md se commitea y se re-implementa en Go en la próxima clase.

🎬 Demo 2 · Spec Kit de punta a punta

De un design.md de juguete a código

claude — ~/todo-api

$

✓ spec.md (requirements + user stories + edge cases)

Una idea en un .md → spec → plan → tareas → código, todo versionado y revisable. La spec manda; el código es el output, no la fuente de verdad.

2.3 · MCPs y Tools

Primero: ¿cómo usa una tool un LLM?

  • Un LLM solo genera texto. Para actuar le das tools: una función + su descripción.
  • El modelo no ejecuta nada: decide qué tool llamar y con qué argumentos. Tu código la corre y le devuelve el resultado.
from langchain_core.tools import tool
from langchain_anthropic import ChatAnthropic

@tool
def get_weather(city: str) -> str:
    """Devuelve el clima actual de una ciudad."""
    return weather_api(city)

llm = ChatAnthropic(model="claude-sonnet-4-6").bind_tools([get_weather])
llm.invoke("¿Qué tiempo hace en Córdoba?")
# → el modelo pide: get_weather("Córdoba")  ·  tu runtime la ejecuta
Definir una tool a mano para cada modelo y framework no escala. MCP estandariza justamente esto → al problema M×N.
Sembrar la intuición: el modelo decide, tu código ejecuta. Esto es lo que MCP estandariza. Puente directo al M×N.

2.3 · MCPs y Tools

El problema M×N

  • 3 modelos × 10 tools = 30 integraciones custom
  • Cada una con su auth, su schema, su error handling
  • Insostenible de mantener
Mismo problema que resuelve un service mesh: en vez de N² conexiones, una abstracción común.
Anclar en service mesh. Ya lo conocen.

2.3 · MCPs y Tools

Model Context Protocol = el USB-C de la IA

  • Anthropic, 25 de noviembre de 2024
  • Inspirado en el LSP de VS Code: write once, plug anywhere
  • Adopción: Anthropic, OpenAI (mar-2025, Altman), Cursor, VS Code, Zed, Replit
  • >10.000 servers públicos activos (Linux Foundation, 9-dic-2025)
La frase USB-C. Un server se escribe una vez y cualquier host lo usa.

2.3 · MCPs y Tools

Las 3 primitivas de MCP

Tools → funciones invocables por el modelo (query DB, mandar email, crear issue)
Resources → data leíble por el modelo (archivos, configs, registros · por URI)
Prompts → templates reutilizables / slash commands

Handshake: initialize → capabilities → tools/list → tools/call

Mapear cada primitiva a algo que ya conocen.

2.3 · MCPs y Tools

Anatomía de un MCP server

Un server = un nombre + una lista de tools tipadas. Cada tool se describe sola.

from fastmcp import FastMCP

mcp = FastMCP("filesystem")

@mcp.tool()
def list_directory(path: str) -> list[str]:
    """Lista los archivos de un directorio."""
    return os.listdir(path)

@mcp.tool()
def read_file(path: str) -> str:
    """Devuelve el contenido de un archivo de texto."""
    return open(path).read()

Estas son las mismas tools (read_file, list_directory…) que vas a ver en la demo de al lado: ahí usamos el server-filesystem oficial, que es exactamente esto pero ya hecho. El modelo lee el nombre, la descripción y los tipos para decidir cuándo llamarla.

Conectar explícito: el server de la demo expone estas mismas tools. El docstring ES lo que el modelo lee para decidir.

2.4 · Claude Skills

Estructura de un Skill

my-skill/
├── SKILL.md          # Required: frontmatter + instructions
├── references/       # Optional: docs cargados on-demand
├── scripts/          # Optional: scripts ejecutables
├── assets/           # Optional: templates, logos
└── evals/            # Recommended: tests
  • Personal: ~/.claude/skills/<nombre>/
  • Proyecto (compartido por el equipo): .claude/skills/<nombre>/
  • Hot-reload: agregás o editás un skill y toma efecto sin reiniciar la sesión
En el repo se commitea .claude/skills/ y todo el equipo lo hereda.

2.4 · Claude Skills

Skill + MCP + API, juntos

1 · Skill → el procedimiento: "para un refund, validá el monto, registrá el motivo y avisá al cliente"
2 · MCP server → expone la tool create_refund(...)
3 · API → Stripe ejecuta el refund de verdad
Pedido del usuario
  └─→ Skill "refunds"          (lee el procedimiento del equipo)
        └─→ MCP tool create_refund(monto, motivo)
              └─→ API de Stripe        →  ✓ refund hecho
El Skill orquesta · el MCP conecta · la API ejecuta.
El skill no "habla" con Stripe; le dice al modelo CÓMO usar la tool de Stripe que el MCP expone.
